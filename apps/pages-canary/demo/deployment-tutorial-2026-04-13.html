Deployment Tutorial

Ship your app to the homelab K3s cluster with ArgoCD

1

Create the Helm Chart

Each application lives in apps/<app-name>/ in the homelab repository. Start with the bjw-s app-template as a base.

apps/my-app/ ├── Chart.yaml # app-template v4.6.2 dependency ├── values.yaml # deployment config (image, ingress, PVC) └── templates/ # optional custom manifests
2

Configure values.yaml

Define your container image, ingress hostname, and any persistent volumes. The app-template handles the boilerplate.

app-template: controllers: main: containers: main: image: repository: 10.0.20.11:32309/my-app tag: "latest" ingress: main: hosts: - host: my-app.eaglepass.io
Tip: Always add cloudflare-proxied: "true" to ingress annotations so external-dns creates proxied DNS records through Cloudflare.
3

Build and Push the Image

Container images are built with Podman and pushed to the private Harbor registry on the homelab network.

podman build -t 10.0.20.11:32309/my-app:latest \ -f Dockerfile . podman push 10.0.20.11:32309/my-app:latest \ --tls-verify=false --quiet
Warning: Never use kubectl apply directly. ArgoCD auto-heals and will revert manual changes. Push to Gitea instead.
4

Commit to Gitea

Push your Helm chart changes to the homelab repo on Gitea. ArgoCD watches this repo and will automatically sync within 1-2 minutes.

git add apps/my-app/ git commit -m "feat: add my-app deployment" git push origin main
5

Verify the Deployment

Check that ArgoCD picked up the change and the pod is running healthy.

kubectl get pods -n my-app kubectl get ingress -n my-app curl -sI https://my-app.eaglepass.io/ | head -3
Tip: If external-dns isn't creating the DNS record, check for stale CNAME conflicts in Cloudflare. A CNAME at the same hostname will block A record creation with error 81054.
